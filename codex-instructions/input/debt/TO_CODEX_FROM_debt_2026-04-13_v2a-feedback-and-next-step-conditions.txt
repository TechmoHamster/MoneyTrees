2. FEEDBACK ITEMS
==================================================

--------------------------------------------------
A. TRUST-STATE PRECISION IS STILL TOO COARSE
--------------------------------------------------

Current trust labeling is directionally correct, but still lacks nuance.

Observed issue:
- payment amount trust defaults heavily to Manual or Custom
- there is still limited distinction between:
  - verified amortized loan payment
  - user-entered estimate
  - derived vs inferred values

Expectation:
Refine trust-state granularity where it can be improved cleanly inside current V2A scope, especially for:
- payment amount reliability
- interest-model confidence
- projection accuracy tiers

Do not invent complexity for its own sake.
But do not leave obviously collapsed categories untouched if they can be improved now.

--------------------------------------------------
B. EXTRA-PAYMENT SCENARIO LABELING IS OVER-SIMPLIFIED
--------------------------------------------------

Observed issue:
All extra-payment scenarios are labeled as Custom regardless of underlying model quality.

Why this matters:
- a no-interest fixed loan with deterministic math should not look identical to a weakly supported custom scenario
- current labeling hides when the scenario is mathematically exact vs user-adjusted

Expectation:
Where feasible inside V2A scope, improve scenario trust expression so future phases can distinguish between:
- exact adjusted scenario
- estimated adjusted scenario
- truly custom scenario

Debt Team is not requiring final naming to be locked now.
But the current flattening should be reviewed and improved if practical.

--------------------------------------------------
C. CREDIT CARD LIMITATION HANDLING MUST STAY STRICT
--------------------------------------------------

Debt Team agrees with the current decision to keep credit-card payoff projections trust-limited in V2A.
That must remain true.

However, this area is the highest-risk area moving into V2B.

Expectation:
- do not loosen trust limits for credit cards prematurely
- do not approximate minimum payment behavior generically
- preserve clear limitation language where the model is intentionally incomplete

If any V2A adjustment touches card projections, it must preserve strict anti-false-precision behavior.

--------------------------------------------------
D. TEST COVERAGE IS BETTER, BUT STILL LIGHT
--------------------------------------------------

The new test file is a good step.
But Debt Team still considers math-layer coverage light.

Missing or still under-covered areas include:
- cadence variation boundaries
- malformed / partial input handling
- projection stability across longer tails
- weak-input failure paths
- payment / interest edge cases

Expectation:
Add any high-value V2A tests that are reasonable within current phase scope.
The goal is not to fully solve all future coverage now.
The goal is to reduce obvious blind spots before V2B builds on this layer.

--------------------------------------------------
E. STATIC PROJECTION ASSUMPTIONS MUST STAY EXPLICIT
--------------------------------------------------

Current projection behavior assumes:
- constant payment
- stable APR
- no behavioral change

Debt Team accepts that for V2A.

Expectation:
Make sure those assumptions remain clearly visible and inspectable wherever the projection is shown.
If assumption clarity can be improved within V2A scope, do it now.

Do not begin dynamic card-behavior work in this pass.
Just make sure static-projection limits are explicit enough that V2B is building on a cleanly labeled foundation.
