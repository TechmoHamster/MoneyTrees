To: Writer
From: Debt Team
Date: 2026-04-11
Subject: Debt Feature Brief for MoneyTrees

Writer —

As of today, Debt is being formed as a new working team for the MoneyTrees product.

This memo is to brief you on the feature direction so it can be translated into clean written product documentation, implementation briefs, and any supporting memos needed for the repo.

==================================================
1. FEATURE PURPOSE
==================================================

Debt should exist to track and calculate debt-related obligations across the product.

Primary debt types in scope:
- loans
- credit cards
- financed purchases
- buy now, pay later plans

The goal is not to create a second advice layer.
The goal is to create a canonical debt model and debt-math engine that feeds the rest of the product cleanly.

==================================================
2. PRODUCT BOUNDARY
==================================================

Debt should own:
- debt-specific input structure
- debt-specific calculation logic
- amortization / payoff schedule logic
- minimum-payment logic where applicable
- payoff projection logic
- debt-specific derived outputs

Debt should NOT own:
- recommendation logic
- historical analysis copy
- overview prioritization logic
- Atlas interpretation language
- Bills table-first workspace behavior

Working architecture:
Debt engine -> Bills -> Overview -> Reporting -> Atlas

==================================================
3. LOANS REQUIREMENTS
==================================================

For loans, the product should allow the user to input:
- lender / account name
- loan type
- original amount
- current balance
- APR / interest rate
- term length
- start date
- payment cadence
- optional extra payment

The product should calculate:
- minimum monthly payment
- amortization schedule
- payoff date
- principal vs interest breakdown
- total projected interest
- remaining balance over time

Debt types to support under loans:
- standard loans
- auto loans
- student loans

Note for writing/spec purposes:
student loans should not be over-described as though every repayment program is modeled in v1.

==================================================
4. CREDIT CARD REQUIREMENTS
==================================================

For credit cards, the product should allow the user to input:
- card name
- issuer
- store-specific flag if relevant
- APR
- credit limit
- current balance
- statement balance
- due date
- promo APR details if present
- deferred-interest flag if present
- fees if relevant

The product should calculate:
- estimated minimum payment
- utilization
- carried-interest estimate
- payoff timeline if only minimum is paid
- payoff timeline with custom payment
- promo expiration exposure

Important product note:
credit card minimum-payment logic should be written/spec’d as issuer-template or rule-based, not as one fake universal formula.

==================================================
5. BNPL / FINANCED PURCHASE REQUIREMENTS
==================================================

For BNPL and financed purchases, the product should allow the user to input:
- merchant
- purchase amount
- finance type
- number of payments or term length
- cadence
- APR or 0% promo rate
- promo end date if relevant
- deferred-interest flag if relevant
- down payment
- fees if relevant

The product should calculate:
- installment amount
- payment schedule
- payoff date
- remaining installments
- promo expiration exposure
- deferred-interest exposure

The writing/spec should clearly distinguish between:
- pay-in-4
- equal installment plans
- 0% promo financing
- interest-bearing financing
- deferred-interest retail financing

==================================================
6. BILLS INTEGRATION
==================================================

Debt must tie into Bills without replacing Bills.

Bills remains the operational workspace and obligation record layer.
Debt should feed structured truth into Bills so users can:
- add a debt-linked bill entry
- select loan / credit card / BNPL / financed purchase
- save debt records cleanly
- view payment breakdowns
- map payments back to the debt account / schedule

==================================================
7. OVERVIEW INTEGRATION
==================================================

Overview should surface factual debt signal only.

Examples of acceptable overview debt signals:
- total active debt
- monthly minimum debt burden
- active BNPL count
- cards above high utilization
- promo plans nearing expiration
- debt share of unpaid obligations
- past-due debt obligations

Overview should not be written as recommendation logic.

==================================================
8. REPORTING INTEGRATION
==================================================

Reporting should analyze debt patterns without becoming advisory.

Good debt reporting candidates:
- total debt balance over time
- monthly minimum burden over time
- interest paid by period
- principal reduction by period
- utilization trends
- debt-type concentration
- newly opened debt by period
- paid-off accounts by period
- promo expiration counts
- debt-related late fees

==================================================
9. ATLAS INTEGRATION
==================================================

Atlas should consume debt truth, not redefine it.

Debt should expose to Atlas:
- balances
- payment schedules
- utilization
- interest burden
- payoff projections
- promo deadlines
- deferred-interest flags
- scenario-ready variables

Atlas can then interpret and recommend using that shared truth.

==================================================
10. RECOMMENDED V1 SHAPE
==================================================

Recommended V1 includes:
- installment loans
- auto loans
- standard student loan support
- credit cards
- BNPL plans
- financed purchases

Mortgage support may be schema-ready but should not necessarily be full V1 unless scope expands.

==================================================
11. WRITER REQUEST
==================================================

Please convert this brief into the following written assets as needed:
- a polished Debt Team operating brief
- a product-facing feature brief
- a clean implementation summary for handoff
- any dependency memo needed for Bills / Overview / Reporting / Atlas

Core framing to preserve:
Debt is a structured debt-truth and debt-math layer.
It should not become a duplicate advice, reporting, or dashboard layer.
