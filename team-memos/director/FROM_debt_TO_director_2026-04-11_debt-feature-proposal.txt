To: Director
From: Debt Team
Date: 2026-04-11
Subject: Debt Feature Proposal for MoneyTrees

Director —

As of today, Debt is being formed as a new working team for the MoneyTrees product.

This memo summarizes the proposed role of the Debt feature and how it should integrate with the rest of the application without colliding with existing section boundaries.

==================================================
1. PROPOSED ROLE OF DEBT
==================================================

Debt should be a canonical debt-truth and debt-math layer.

It should exist to model and calculate:
- loans
- credit cards
- financed purchases
- buy now, pay later plans

Debt should not become:
- a second dashboard layer
- a recommendation engine
- a duplicate reporting surface
- an Atlas clone

Recommended architecture:
Debt engine -> Bills -> Overview -> Reporting -> Atlas

==================================================
2. CORE PRODUCT PURPOSE
==================================================

Debt should allow users to input key debt variables and immediately generate the math and schedule logic needed to track payoff, cost, and timing.

The product should support:
- installment loan calculations
- auto loan calculations
- standard student loan calculations
- credit card minimum-payment tracking
- financed purchase tracking
- BNPL plan tracking

==================================================
3. LOANS
==================================================

Debt should support input of:
- lender / account name
- loan type
- original amount
- current balance
- APR
- term length
- start date
- payment cadence
- optional extra payment

Debt should calculate:
- minimum monthly payment
- amortization schedule
- payoff date
- principal vs interest breakdown
- total projected interest
- remaining balance over time

==================================================
4. CREDIT CARDS
==================================================

Debt should support input of:
- card name
- issuer
- store card flag where relevant
- APR
- credit limit
- current balance
- statement balance
- due date
- promo APR details
- deferred-interest flag
- relevant fees

Debt should calculate:
- estimated minimum payment
- utilization
- carried-interest estimate
- payoff timeline using minimum only
- payoff timeline using custom payment
- promo expiration exposure

Product note:
credit card minimum payments should be modeled with issuer-template or rule-based logic, not a fake one-size-fits-all formula.

==================================================
5. BNPL / FINANCED PURCHASES
==================================================

Debt should support input of:
- merchant
- purchase amount
- finance type
- number of payments or term length
- cadence
- APR or 0% promo rate
- promo end date
- deferred-interest flag
- down payment
- fees

Debt should calculate:
- installment amount
- payment schedule
- payoff date
- remaining installments
- promo expiration exposure
- deferred-interest exposure

The system should distinguish between:
- pay-in-4
- equal installment plans
- 0% promo financing
- interest-bearing financing
- deferred-interest retail financing

==================================================
6. INTEGRATION WITH BILLS
==================================================

Debt should feed Bills, not replace Bills.

Bills remains the operational table-first workspace and obligation record layer.

Debt integration should allow users to:
- create a debt-linked bill entry
- select loan / credit card / BNPL / financed purchase
- save debt records cleanly
- inspect payment breakdowns
- map payments back to debt accounts and schedules

==================================================
7. INTEGRATION WITH OVERVIEW
==================================================

Overview should surface factual debt signal only.

Useful overview-level debt metrics include:
- total active debt
- monthly minimum debt burden
- active BNPL count
- high-utilization cards
- promo plans nearing expiration
- debt share of unpaid obligations
- past-due debt obligations

==================================================
8. INTEGRATION WITH REPORTING
==================================================

Reporting should analyze debt patterns without becoming advisory.

Useful reporting metrics include:
- total debt balance over time
- monthly minimum burden over time
- interest paid by period
- principal reduction by period
- utilization trends
- debt-type concentration
- new debt opened by period
- paid-off debt accounts by period
- promo expiration counts
- debt-related late fees

==================================================
9. INTEGRATION WITH ATLAS
==================================================

Atlas should consume debt truth, not redefine it.

Debt should expose to Atlas:
- balances
- payment schedules
- utilization
- interest burden
- payoff projections
- promo deadlines
- deferred-interest flags
- scenario-ready variables

That allows Atlas to interpret and recommend using shared truth rather than local debt logic.

==================================================
10. RECOMMENDED V1 SHAPE
==================================================

Recommended V1 includes:
- installment loans
- auto loans
- standard student loan support
- credit cards
- BNPL plans
- financed purchases

Mortgage support may be schema-ready, but does not need to be full V1 unless scope expands.

==================================================
11. STRUCTURAL NOTE
==================================================

Current repo communication rules do not yet formally list Debt or Writer as team-memo lanes.

That should be updated so the repository structure matches the current team model.

— Debt Team
