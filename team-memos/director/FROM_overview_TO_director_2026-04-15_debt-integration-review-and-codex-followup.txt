==================================================

Two issues remain.

A) Routing is still too generic
Most click actions route to the Debt section broadly rather than to the most specific Debt destination available.

This weakens the implementation versus the intended routing behavior.

Examples of preferred refinement:
- Accounts Behind -> Debt accounts filtered to late/delinquent if supported
- Debt Cash Needed -> Debt cash-window or schedule view if supported
- Accounts Requiring Review -> Debt activity or review-filtered accounts if supported

B) Adapter-layer derivation is still heavier than ideal
The implementation still computes some compact Overview values in the adapter layer rather than consuming only fully explicit Debt-owned summary outputs end-to-end.

This is not a major boundary violation, but it is weaker than the desired field-contract purity and could drift later if Debt rules change.
