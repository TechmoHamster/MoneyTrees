Reporting Team —

Debt V2.2 has now been reviewed and approved by Debt Team.

Debt is now a type-specific intake and lifecycle-aware debt engine. That means Reporting has new structured Debt facts available for analytical use.

This memo defines what Reporting should consume, what Reporting should analyze, and what Reporting must not take ownership of.

==================================================
1. CONTEXT
==================================================

Debt V2.2 now provides richer debt data, including:
- debt-type-specific account facts
- standing state
- arrangement overlays
- lifecycle events
- term versions
- continuity events
- source-quality / trust labels
- failed and reversed payment events
- partial payment and cure facts
- days past due
- amount needed to cure
- next escalation date
- collections / settlement / charged-off facts
- capitalization events
- cash-window outputs
- source conflicts
- confidence / trust states

Debt remains the owner of debt truth, debt math, lifecycle state, schedules, and debt-specific outputs.

Reporting should analyze these outputs over time.
Reporting must not recreate Debt logic.

==================================================
2. REPORTING BOUNDARY REMINDER
==================================================

Per Reporting operating rules, Reporting is the product analysis layer.

Reporting explains:
- what changed
- compared to what
- how the selected period looks
- where concentration or pattern shifts exist
- how system pressure/risk outputs changed over time

Reporting is not:
- a decision layer
- a recommendation layer
- a prioritization layer
- a scoring layer
- a second Overview
- a lightweight Atlas clone

Core rule:
Observed change -> comparison baseline -> analytical meaning.
Stop there.

5. COPY AND INSIGHT RULES
==================================================

Reporting should keep copy analytical and neutral.

Good examples:
- Debt balance increased by $420 versus the previous period, mostly from credit-card balance growth.
- Failed debt payments rose from 0 to 2 in the selected period, showing weaker payment completion during this window.
- Three accounts were marked verification-needed, which limits confidence in some debt projections.
- Debt cash required in the next 30 days increased by $180 versus the prior window.

Bad examples:
- Pay these debts first.
- This is your top priority.
- You should fix the delinquent accounts now.
- Attack the highest-interest account first.
- Focus on credit cards before anything else.

Reporting may show contributors.
Reporting may show concentration.
Reporting may show change.
Reporting may not recommend actions.

==================================================
6. TIME / COMPARISON REQUIREMENTS
==================================================

Debt analytics must obey Reporting's period rules.

Every Debt reporting module must clearly respect:
- selected reporting range
- comparison window
- KPI deltas
- chart range
- insight copy
- export range
- drill-down context

If a debt event falls outside the selected period, do not count it inside the period.

If a debt value is current-state only and not historical, label it as current-state rather than pretending it is a period trend.

This is important for Debt because some outputs are snapshots and some are event history.
Do not blend those without labeling the difference.

==================================================
7. DRILL-DOWN / EXPORT EXPECTATIONS
==================================================

Reporting should support drill-down into Debt-owned records without taking ownership of the records.

Drill-down should allow the user to inspect:
- account detail
- lifecycle events
- payment events
- failed / reversed payment events
- arrangement overlays
- continuity chain facts
- trust / source-quality facts

Exports should include Debt reporting fields only when they are visible or selected in the current Reporting context.

Exports must preserve:
- active range
- comparison range where relevant
- source labels
- confidence labels
- Debt-owned field names

==================================================
8. WHAT REPORTING MUST NOT DO
==================================================

Do not:
- recreate Debt payoff math
- recreate standing-state logic
- recreate cure logic
- create local delinquency thresholds
- create local debt risk scores
- create local priority rankings
- create debt recommendations
- re-label Debt severity or standing states
- flatten arrangement overlays into vague risk language
- treat generic debt pressure as advice
- create Atlas-style next-step cards

Reporting should never say what the user should do.
Reporting should show what changed and why it matters analytically.
