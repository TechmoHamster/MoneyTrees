==================================================
4. SNAPSHOT VS EVENT-HISTORY HANDLING
==================================================

Reporting will explicitly separate:
- snapshot facts
- event-history facts

Working rule:
- snapshot fields will be labeled as current-state or point-in-time facts when historical series is not available
- event-history fields will only be counted/trended inside the selected period when an actual event timestamp exists in-range
- Reporting will not present current-state-only values as if they are period trends
- Reporting will distinguish “current balance” style reads from “balance changed by X during this period” reads

If a Debt output is only available as a current snapshot:
- Reporting may show it as current-state context
- Reporting will not fake a trend line or period delta from it

==================================================
5. STALE / LIMITED / SOURCE-CONFLICT DATA DISPLAY
==================================================

Reporting will treat trust and data-quality states as analytical context, not recommendation triggers.

Planned handling:
- expose stale / limited / conflict counts as neutral KPI or support metrics where useful
- show trust/data-quality modules or notes when those signals materially affect interpretation
- label affected outputs clearly rather than hiding the limitation
- avoid strong analytical claims when Debt marks the underlying data as stale, limited-confidence, or conflicting
- preserve Debt-owned labels in drill-down and export paths

Copy style example:
- Three debt accounts were marked verification-needed in this window, which limits confidence in some debt reporting outputs.

==================================================
6. DRILL-DOWN BEHAVIOR
==================================================

Reporting drill-down should route into Debt-owned record detail without taking ownership of the record.

Planned drill-down targets include:
- account detail
- lifecycle events
- payment events
- failed / reversed payment events
- arrangement overlays
- continuity chain facts
- trust / source-quality facts

Rule:
Reporting summarizes and routes.
Debt owns the underlying detail view and record truth.

==================================================
7. EXPORT BEHAVIOR
==================================================

Reporting exports should include Debt reporting fields only when those fields are part of the active reporting context.

Planned export rules:
- preserve selected reporting range
- preserve comparison range where relevant
- preserve Debt-owned field names where exposed
- preserve source / confidence / trust labels where relevant to the visible reporting view
- avoid mixing snapshot-only and event-history fields without labeling the distinction
- include only visible / selected debt-reporting context rather than dumping raw Debt internals by default

==================================================
8. BLOCKERS / DATA-CONTRACT QUESTIONS
==================================================

Current questions for Debt / shared contract clarification:

1. Which Debt outputs are guaranteed historical series versus current-state-only snapshots?
2. Which timestamp should Reporting treat as canonical for lifecycle/event trending when multiple timestamps exist?
3. Which projected-interest / payoff-confidence fields are safe for period-over-period analytical comparison versus current-state display only?
4. Which Debt trust labels should be treated as user-visible by default versus drill-down only?
5. For continuity events, what is the preferred analytical contract for predecessor/successor rollups without double counting?
6. Are debt cash-window outputs intended to be point-in-time snapshots, recalculated range outputs, or both?
7. Which Debt-owned labels are stable enough to expose directly in exports without future rename risk?
